# 星盘生成与解析调用说明

## 1. 这个项目是如何“根据出生信息生成星盘”的

本项目的生成链路可以分成 4 步：

1. 前端收集出生信息
2. 后端调用 `jyotishganit` 计算原始吠陀星盘
3. 后端把原始星盘整理成统一结构化 JSON
4. 前端把结构化 JSON 渲染成页面，并继续调用 AI 解读与 AI 校时

项目里的主入口分别是：

- 前端页面：[frontend/app/natal/page.tsx](/D:/chiyu/ai印度占星/frontend/app/natal/page.tsx)
- 后端接口：[backend/app/main.py](/D:/chiyu/ai印度占星/backend/app/main.py)
- 星盘计算与结构化映射：[backend/app/chart_service.py](/D:/chiyu/ai印度占星/backend/app/chart_service.py)
- 请求模型：[backend/app/schemas.py](/D:/chiyu/ai印度占星/backend/app/schemas.py)

## 2. 前端输入了哪些信息

用户在 `/natal` 页面输入或选择这些字段：

- 姓名 `name`
- 出生日期 `birth_date`
- 出生时间 `birth_time`
- 出生地点文本 `locationName`
- 纬度 `latitude`
- 经度 `longitude`
- 时区 `timezone_offset`

其中“出生地点”这一项有两种辅助方式：

- 通过 `Nominatim` 文本搜索地点候选
- 在世界地图上直接点击坐标

前端最终真正提交给后端生成星盘的，是这 6 个核心计算字段：

```json
{
  "name": "Test User",
  "birth_date": "1996-07-04",
  "birth_time": "09:10",
  "latitude": 31.2304,
  "longitude": 121.4737,
  "timezone_offset": 8
}
```

对应后端请求模型在：

## 3. 后端是如何生成星盘的

后端的星盘接口是：

- `POST /api/v1/birth-chart`

接口定义在：

- [backend/app/main.py](/D:/chiyu/ai印度占星/backend/app/main.py:31)

### 3.1 计算主流程

后端收到出生信息后，会进入：

- `calculate_birth_chart(payload)`

位置：

- [backend/app/chart_service.py](/D:/chiyu/ai印度占星/backend/app/chart_service.py:80)

核心逻辑是：

1. 把 `birth_date + birth_time` 合并成一个 `datetime`
2. 优先调用 `jyotishganit.calculate_birth_chart(...)`
3. 再调用 `jyotishganit.get_birth_chart_json(chart)` 拿到原始 JSON
4. 对原始 JSON 做二次提取与补全
5. 返回适合前端和 AI 使用的统一结构

### 3.2 实际依赖的计算引擎

项目当前依赖的真实占星计算引擎是：

- [`jyotishganit`](https://github.com/northtara/jyotishganit)

它负责底层天文与吠陀占星计算，例如：

- D1 本命盘
- Panchanga
- Vimshottari Dasha
- 多个 divisional charts
- Ashtakavarga
- 行星位置、宫位、nakshatra、pada、部分 shadbala 等

### 3.3 如果引擎调用失败会怎样

如果 `jyotishganit` 抛错，项目不会直接失败，而是进入 fallback：

- `_calculate_fallback(...)`

位置：

- [backend/app/chart_service.py](/D:/chiyu/ai印度占星/backend/app/chart_service.py:155)

这个 fallback 主要用于：

- 流程联调
- UI 演示
- 在本地引擎不可用时保证页面还能跑通

但 fallback 数据不代表严肃占星结果。

## 5. 为什么有些字段会出现“未提供”

这类情况通常有 3 种原因：

### 5.1 引擎原始返回里没有该字段

例如有些宫位原始 JSON 没带完整 cusp 信息，项目后面补了自动推导逻辑。

### 5.2 该字段对某些对象本来就不稳定或不常规

最典型的是：

- `Rahu`
- `Ketu`

它们是影子行星，不是所有传统体系都把它们按普通行星一样计算 `Shadbala`。因此项目现在对这两项的展示策略更保守：

- 不伪造数值
- 页面会解释为“传统上通常不适用/不计入六力”

### 5.3 这是 fallback 演示数据

如果当前返回的是 fallback，不代表真实专业占星结果。

## 7. AI 是如何“解析这张盘”的

项目里有两条 AI 调用链：

1. 星盘解读
2. 出生时间校正

### 7.1 星盘解读

接口：

- `POST /api/v1/chart-reading`

后端入口：

- [backend/app/main.py](/D:/chiyu/ai印度占星/backend/app/main.py:36)

前端调用：

- [frontend/app/natal/page.tsx](/D:/chiyu/ai印度占星/frontend/app/natal/page.tsx:624)

前端会把这些内容传给后端：

```json
{
  "focus": "core",
  "question": "请基于这张印度星盘做一份整体解读……",
  "location_name": "Shanghai, China",
  "chart": { "...完整结构化星盘..." }
}
```

后端再把结构化星盘整理成一段“事实文本”，供 AI 解读服务使用。

相关逻辑在：

- [backend/app/chart_reading_service.py](/D:/chiyu/ai印度占星/backend/app/chart_reading_service.py)

### 7.2 出生时间校正

接口：

- `POST /api/v1/birth-time-rectifier`

后端入口：

- [backend/app/main.py](/D:/chiyu/ai印度占星/backend/app/main.py:41)

前端调用：

- [frontend/app/natal/page.tsx](/D:/chiyu/ai印度占星/frontend/app/natal/page.tsx:657)

请求结构：

```json
{
  "location_name": "Shanghai, China",
  "chart": { "...完整结构化星盘..." },
  "life_events": [
    {
      "title": "离家读大学",
      "date_hint": "2014-09",
      "description": "第一次长期离开原生家庭。"
    }
  ]
}
```

也就是说，AI 校时不是“重新算星盘”，而是：

1. 先基于已有出生时间生成一张盘
2. 再结合人生事件，让 AI 判断这个时间是否需要微调

## 8. 如何直接调用这些接口

### 8.1 生成星盘

```bash
curl -X POST "http://127.0.0.1:8000/api/v1/birth-chart" ^
  -H "Content-Type: application/json" ^
  -d "{\"name\":\"Test User\",\"birth_date\":\"1996-07-04\",\"birth_time\":\"09:10\",\"latitude\":31.2304,\"longitude\":121.4737,\"timezone_offset\":8}"
```

### 8.2 请求 AI 解读

```bash
curl -X POST "http://127.0.0.1:8000/api/v1/chart-reading" ^
  -H "Content-Type: application/json" ^
  -d "{\"focus\":\"core\",\"question\":\"请做整体解读\",\"location_name\":\"Shanghai, China\",\"chart\":{...}}"
```

### 8.3 请求 AI 校时

```bash
curl -X POST "http://127.0.0.1:8000/api/v1/birth-time-rectifier" ^
  -H "Content-Type: application/json" ^
  -d "{\"location_name\":\"Shanghai, China\",\"chart\":{...},\"life_events\":[...]}"
```

## 9. 当前这套设计的优点

- 底层计算和页面展示解耦
- 结构化 JSON 可以复用给页面、AI 解读、AI 校时
- 引擎原始返回保留在 `chart.raw`，后续可继续扩字段
- 即使本地引擎失败，也有 fallback 保障 UI 可运行

## 10. 如果后面还要继续扩展，最建议补哪几项

建议优先级如下：

1. 把 `dasha.upcoming` 做成时间轴组件
2. 给每个分盘做独立详情页或折叠面板
3. 把 `Ashtakavarga` 从表格升级成热力图
4. 增加 `raw` 到“标准化字段”的映射文档，方便以后继续补字段
5. 如果要做 API 给外部系统调用，可以单独再拆一层“轻量响应”和“专业响应”

---

## 11. 一句话总结

这个项目的核心不是“AI 直接生成星盘”，而是：

- 先由 `jyotishganit` 根据出生日期、时间、经纬度、时区计算出原始吠陀星盘
- 再由后端把原始星盘解析成统一结构化数据
- 最后由前端展示，并把同一份结构化数据继续交给 AI 做解读和校时
